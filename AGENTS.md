# AGENTS.md — инструкции для ИИ-ассистентов

## Основные правила работы

- Если задачу можно решить без кода — расскажи как, не пиши код
- Код меняем только в том случае, если без этого не получить результат
- Если пользователь предлагает сложную правку — попробуй упростить, если это возможно, и предложи решение
- Если пользователь что-то упускает или не прав — укажи на проблему и предложи решение
- KISS и DRY — главные принципы
- Основная цель — краткий и максимально читаемый код с хорошим форматированием
- Старайся не просто показать решение, а объяснить, почему именно это решение выбрано и в чём его преимущество перед альтернативами
- После завершения фичи (фразы «готово», «заработало», «всё», «молодец») — спросить, не пора ли обновить SUMMARY.md
- Если возникла проблема, потребовавшая 3+ итераций исправления — зафиксировать причину и решение в AUTOPILOT.md (Фаза 8) и CHANGELOG.md
- Если в проекте есть `AUTOPILOT.md` — следуй ему автономно. Один запрос разрешений в Фазе 0, дальше без подтверждений. Цикл: реализация → тесты → исправление → тесты → обновление autopilot_state.json → следующий пункт

## Особенности (ключевые правила)

- Пользователь (разработчик проекта) не знает английского. Все логи, UI, комментарии, оповещения — на русском языке. Исключение: общепринятые названия (rust, piper, egui, torch и т.п.), названия файлов и технические идентификаторы.

- **Никогда не хардкодить токены/ключи в исходники.**

- **Версию менять только в `Cargo.toml`** (`package.version`). Код должен получать номер версии как переменную (`env!("CARGO_PKG_VERSION")`), а не хардкодом.

- **Многоязычность с нуля.** Сразу на стадии формирования архитектуры предусматривать папку `locales/` с json-файлами локалей. Проект должен учитывать склонения языковых групп:
  
  - `одно яйцо, два яйца, много яиц`
  - `работать с настройкой, взять из настроек, сохранить в настройках`

- **Автосохранение настроек.** Все имеющиеся и будущие настройки должны автосохраняться при изменении и автовосстанавливаться после загрузки/перезагрузки проекта.

- **Кодировка: UTF-8 без BOM.** Весь обмен данными между Rust и текстовыми файлами (включая python, bat (cmd), ps1, json и другие взаимодействия) — только UTF-8 без BOM. На русской Windows `sys.stdin.encoding` по умолчанию cp1251 — никогда на него не полагаться.

- **Локальное время.** При работе с датой-временем (кэш, логи, формирование файлов) использовать локальное время: `chrono::Local::now()`.

- **Файл быстрого запуска.** Создать `___run.bat` в корне проекта с прекомпиляцией и запуском. Сообщения на латинице (cmd не рендерит кириллицу):
  
  ```batch
  @echo off
  chcp 65001 >nul 2>&1
  cd /d "%~dp0"
  echo [1/2] Sborka...
  cargo build
  if %errorlevel% neq 0 (
      echo [ERROR] Sborka ne udalas!
      pause
      exit /b 1
  )
  echo Zapusk...
  target\debug\project.exe
  pause
  ```

## Нейминг и назначения

### Переменные, классы, функции

- Имя должно отражать **назначение**, а не значение:
  - ✅ `buttonStart`, `textPrimary`, `sliderTrack`
  - ❌ `buttonBlue`, `textDark`, `sliderGrey`
- Каждая переменная хранит собственное значение, а не ссылку на другую:
  - ✅ `textPrimary -> "#E0EDFF"`, `editorText -> "#E0EDFF"`
  - ❌ `editorText -> textPrimary`
- Взаимные ссылки — только по явному запросу разработчика

### Пути

- Не хардкодить пути. Только `%USERPROFILE%`, `%LOCALAPPDATA%`, `%APPDATA%` или `dirs` крейт (загрузи `rust-single-source`)
- Понимать, что платформы могут быть разные. Не привязываться к конкретному диску/пользователю

### Списки

- Если элементов в списке больше 3 — предлагать разработчику вынести в отдельный JSON

## Нейминг и размер файлов

### Нейминг

- Формат: `snake_case`:
  - ✅ `tab_status.rs`, `tab_settings.rs`, `tab_logs.rs`
  - ❌ `status_tab.rs`, `settings_tab.rs`, `logs_tab.rs`
- Имя не должно быть длинным. Если контекст укладывается в одно слово — использовать одно слово
- Для группы однотипных файлов: сначала общий контекст, затем специализация
- ✅ `audio/` подмодули: `tts.rs`, `speech.rs` (контекст — директория)

### Уникальность имён

- **Запрещены файлы-двойники** с одинаковым именем во всём проекте (без учёта пути):
  - ❌ `src/ui/audio.rs` + `src/data/audio.rs`
  - Если имя занято — отразить различие контекстов: `{context}/{context}_helpers.rs`
- Исключение: `mod.rs`, `main.rs`, `lib.rs` — служебные имена, продиктованные языком

### Размер файлов

- **Максимум по умолчанию:** 250 строк на файл.
- **Допустимое превышение:** до +50% (т.е. до 375 строк), если:
  - контекст файла един и целостен,
  - код не подразумевает переиспользование из других мест,
  - и при этом **не** создаются вспомогательные файлы.
- Если для файла понадобились хелперы — правило +50% **не действует**: все файлы группы ≤ 250 строк каждый.

### Механизм при превышении 375 строк

1. **Шаг 1 — хелперы.** Вынести чистые функции в `{context}_helpers.rs`. Все файлы группы ≤ 250 строк.
2. **Шаг 2 — директория.** Превратить в папку `{context}/`:
   - `mod.rs` — публичное API и реэкспорты
   - `render.rs` — отрисовка UI
   - `handlers.rs` — обработчики событий/команд
   - `helpers.rs` — внутренние утилиты
   - Каждый файл ≤ 250 строк.
3. **Шаг 3 — переиспользование.** Если хелперы нужны извне — переместить на уровень выше.

## GitHub

- **Пуш на GitHub только после явного подтверждения пользователя.** Никогда не пушить без разрешения.
- Если пользователь давно не пушил — напомнить и спросить, но не пушить самому.
- Токен: `%USERPROFILE%\.github_token` (ограничен по времени).
- **PowerShell и кириллица:** `Invoke-RestMethod` кодирует тело в Windows-1251 — кириллица ломается. Для публикации релизов с кириллицей использовать `gh` CLI через `cmd.exe`:
  
  ```powershell
  # Тело записать в файл через Write tool (чистый UTF-8 без BOM)
  cmd /c "gh release edit v1.0 --repo user/repo --notes-file C:\path\to\body.md"
  ```

## При описании структуры проекта

Придерживаться порядка:

1. Скрытые папки (с точкой) — алфавитный порядок
2. Обычные папки — алфавитный порядок
3. Скрытые файлы (с точкой) — алфавитный порядок
4. Обычные файлы — алфавитный порядок
5. Для вложенных папок/файлов — тот же принцип рекурсивно
6. Содержимое скрытых папок не расписывать — указать только саму папку
